CRITICAL EXPORT SCHEMA AND ERROR-HANDLING REQUIREMENTS

Design the export layer so that it is genuinely suitable for ChatGPT custom GPT knowledge ingestion, embedding pipelines, and later reprocessing. Nothing should be silently omitted. If any attachment, body, or field cannot be extracted, that failure must be preserved in the export in a clear, structured way.

PRIMARY RULE
Every exported email record must contain:
1. core email metadata
2. normalized email body content
3. all attachment metadata
4. extracted attachment content where readable
5. explicit error and warning records where content could not be extracted
6. a merged AI-ready content field for ingestion
7. provenance fields linking all content back to the source email and attachment

DO NOT silently skip unreadable, encrypted, corrupted, or unsupported files.
If extraction fails, the export must explicitly state:
- what file failed
- why it failed
- whether the failure is likely due to encryption/security/protection/corruption/unsupported type/size limit/OCR limitation
- whether retry may succeed after user action

REQUIRED EXPORT FORMATS

Implement these 3 export outputs:

1. full_export.json
- structured hierarchical export of emails plus attachments

2. ai_ingestion.jsonl
- one ingestion-ready object per email or email-content chunk
- attachment content included in normalized form

3. export_manifest.json
- summary of everything exported, extracted, skipped, failed, unsupported, and partially processed

MANDATORY FAILURE CATEGORIES

Every failed or partial attachment extraction must classify into one of these categories:

- unsupported_file_type
- encrypted_or_password_protected
- corrupt_or_malformed
- empty_file
- too_large
- extraction_timeout
- parser_error
- ocr_failed
- network_fetch_error
- permission_error
- unknown_error

MANDATORY USER-ACTION MESSAGES

Where possible, provide an actionable message for the user.

Examples:
- "Password protected PDF could not be opened. Remove the password or export an unsecured copy and rerun."
- "Scanned image quality too poor for OCR. Provide a cleaner scan or higher-resolution image."
- "Spreadsheet is corrupted or unreadable. Open and resave the file in Excel, then rerun."
- "Attachment type .msg is not yet supported for text extraction. Raw metadata exported only."
- "Attachment exceeded configured size limit and was skipped. Increase max size or process manually."

MANDATORY LOGGING REQUIREMENTS

Build a proper processing log system. Do not rely on vague console messages only.

The system must produce:
1. UI-visible extraction status per email and per attachment
2. downloadable processing log file
3. structured error objects in export output
4. server-side logs with enough detail for debugging

For each processing event, log:
- timestamp
- message_id
- attachment_id if applicable
- filename if applicable
- event_type
- status
- extraction_method
- duration_ms
- error_category
- error_message
- user_action_needed

Suggested event types:
- email_fetch_started
- email_fetch_completed
- attachment_detected
- attachment_download_started
- attachment_download_completed
- attachment_extraction_started
- attachment_extraction_completed
- attachment_extraction_failed
- export_record_written
- export_completed

MANDATORY SECURE/PROTECTED FILE HANDLING

If a file cannot be read due to security/protection:
- do not fail silently
- do not mark it as generic parser failure if you can detect protection
- classify it specifically as encrypted_or_password_protected
- include filename and attachment metadata
- include clear suggested next step:
  "Remove security/password restrictions and rerun extraction."

MANDATORY CONTENT MERGING RULE

For AI use, every exported email must contain a merged text field that combines:
1. normalized email body
2. extracted attachment text from all readable attachments
3. clear separators between sections

Example structure of combined_text:

EMAIL SUBJECT: ...
FROM: ...
TO: ...
DATE: ...

EMAIL BODY:
...

ATTACHMENT: contract.pdf
TYPE: application/pdf
EXTRACTION STATUS: success
CONTENT:
...

ATTACHMENT: secure-financials.xlsx
TYPE: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
EXTRACTION STATUS: failed
REASON: Password protected or encrypted spreadsheet could not be read.
USER ACTION: Remove file protection and rerun extraction.

This combined_text must never omit attachment failures. Even unreadable files must be represented in text form as failure notices so the downstream AI context reflects that something was attached but unreadable.

MANDATORY NO-SILENT-LOSS RULE

No email or attachment content may be silently dropped.
If something cannot be read, exported output must still preserve:
- metadata
- failure category
- reason
- user action required
